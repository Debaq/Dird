Plain language

Explain it simply

What DIRD+ does, why it matters, and how it's used — in clear language, no jargon.

1

What is this?

An application that looks at fundus photographs of your eye and detects signs of damage caused by diabetes. The AI marks the lesions; the clinician reviews and decides.

2

Why does it matter?

Diabetes can damage the retina without the patient noticing until vision is already affected. Catching it early prevents the leading cause of preventable blindness in working-age adults.

3

Who uses it?

Healthcare professionals: ophthalmology, optometry, general practice, primary care. The AI assists, it does not diagnose on its own — the clinician validates every finding before the report.

4

Is my data safe?

Yes. All analysis runs on the clinician's device. Images are never uploaded to the cloud or to any external server. Zero patient data exposure to the internet.

5

How good is it?

Externally validated on 3,662 images the model had never seen: correctly detects 97 out of 100 cases with retinopathy and correctly rules out 93 out of 100 healthy eyes.

Platform

Assisted detection for ophthalmology

YOLO model trained on fundus images, run locally, with clinical reports generated on-device.

Multiclass detection

Optic disc, fovea, hemorrhages, hard and soft exudates, microhemorrhages, and edema. 11 classes defined, 7 active.

Local AI, no cloud

ONNX model running in the browser via onnxruntime-web or as a Tauri binary. Patient data never leaves the device.

Clinical reports

Automatic PDF report generation with findings, metrics, and visualization. Pluggable clinical-guidelines engine.

Fast screening

Inference in seconds per image. Designed for high-volume screening workflows in primary care and telemedicine.

Data sovereignty

Zero transfer to external servers. Aligns with strict privacy principles and local healthcare regulation.

Cross-platform architecture

Tauri desktop (Windows/Linux) + static web build. Same ONNX model in both environments, same experience.

Detection on fundus

Boxes and scores on the fundus image

The model marks each lesion with a color according to type (hemorrhage, exudate, microhemorrhage) and severity. The clinician reviews, corrects, and validates before generating the report.

Reports

Decision-support report — the clinician decides

Editable PDF template with the analyzed image, segmented findings, and severity suggested by the model. The clinician reviews, adjusts as needed, and signs. The AI provides an organized format; diagnostic judgment and the final report always remain with the clinician.

Clinical workflow

From image to screening in minutes

Fundus upload, local inference, clinician review, validation of findings, and report generation. All without opening the browser to the internet.

PCT_fpr02 — recommended operating point

How we calibrated DIRD+ v2 on APTOS 2019 (n=3,662) and why this point is the default.

0 · Context — what problem we solve

DIRD+ is designed primarily as a human-in-the-loop (HITL) tool: a medical technologist or ophthalmologist works with the model, not replaced by it. The model flags, suggests and prioritizes; the human reviews, validates and decides. As a secondary mode, it can run as autonomous screening when context requires it (rural posts, mass screening, no specialist on site).

This validation evaluates the autonomous mode — the model alone, unassisted — because that is the setup measurable against APTOS labels. It is deliberately the most demanding scenario: the worst case from the product's standpoint, since the real use case (HITL) adds clinical judgment on top of the model output. The operational question answered here is the binary screening one: does this image need an ophthalmologist's attention, yes or no?

To validate that we used APTOS 2019 Blindness Detection (Kaggle, n=3,662 images), labeled with the International Clinical Diabetic Retinopathy (ICDR) scale, grades 0 to 4:

ICDR gradeMeaningBinary labeln
0No retinopathyNORMAL1,805
1Mild DRPATHOLOGICAL370
2Moderate DRPATHOLOGICAL999
3Severe DRPATHOLOGICAL193
4Proliferative DRPATHOLOGICAL295

We binarize the ground truth: grade 0 → NORMAL, grades 1-4 → PATHOLOGICAL (1,805 healthy vs 1,857 pathological, 49.3% / 50.7% — balanced). The validation task reduces to "given the detector output, which confidence thresholds best separate healthy from pathological?". That is the question the whole experiment chain pursues — and PCT_fpr02 is the calibrated answer.

Design note: the model is a lesion detector (6 classes: hard_exudate, hemorrhage, cotton_wool_spot, microhemorrhages, optic_disc, fovea), not a grade classifier. The binary rule is: if at least one lesion fires above its threshold, the image is flagged pathological. Anatomical classes (optic disc, fovea) are deliberately ignored because they appear in every retina.

1 · What is PCT_fpr02?

PCT = Per-Class Thresholds. FPR02 = False Positive Rate ≤ 2% measured on normal fundi.

Instead of applying the same confidence threshold to every lesion class (e.g. uniform 0.25), each lesion has its own threshold, calibrated so that the false positive rate on healthy retinas stays below 2% per class.

hard_exudateτ = 0.63
microhemorrhagesτ = 0.26
cotton_wool_spotτ = 0.85
hemorrhageτ = 0.05

Decision rule: ABNORMAL ⇔ ∃ class c with score(c) ≥ τ[c]. If only anatomical landmarks (optic disc, fovea) or nothing is detected, the image is NORMAL.

Why it matters: each lesion class has a different score distribution. microhemorrhages is the model's strongest class (AUC 0.988) but flooded false positives at τ=0.25 because its distribution starts lower. hard_exudate is the opposite: it over-fires on normals and needs a high τ. Per-class calibration unlocks each detector's actual capacity.

2 · How did we get there?

It wasn't an a-priori choice. Six chained experiments, each answering the question the previous one left open:

Binary baseline — does it generalize out-of-domain?

Question: trained on IDRiD; does it work on APTOS without retraining?

Yes, generalizes well (AUC 0.95). But at uniform conf=0.25 it produces 981 false positives on 1,805 normals. Useful for screening, not for production.

Sens 0.997 · Spec 0.456 · MCC 0.541

conf/N sweep — what is the best uniform threshold?

Question: does raising the threshold or requiring multiple detections fix it?

Best uniform point: R3: conf≥0.40 ∧ N≥2. We noticed something: using lesion count as score discriminated better (AUC 0.964) than max score (0.949). A hint that classes contributed heterogeneous signal.

R3 → Sens 0.932 · Spec 0.918 · MCC 0.850

Per-class calibration — what if each lesion gets its own threshold?

We computed ROC curves per class against the binary ground truth and picked τ with three criteria: Youden J, FPR≤5%, FPR≤2%. FPR≤2% won: MCC jumped from 0.85 to 0.913 without sacrificing sensitivity. PCT_fpr02 was born here.

PCT_fpr02 → Sens 0.978 · Spec 0.933 · MCC 0.913

5-fold cross-validation — overfitting or stable?

Question: would those τ work on another sample, or did we memorize APTOS?

Stable. Max coefficient of variation: 8% (one τ only). Rest <1%. Reproducible MCC 0.911 ± 0.015 across folds.

CV: MCC 0.911 ± 0.015 · σ Spec ≤ 0.004

Area filter — are FPs small bboxes?

Partially true. Filtering bboxes <800 px² lifted R3 MCC (0.85 → 0.88), but still below PCT_fpr02. Per-class calibration already discards low-quality detections in a score-driven way.

R3+area800 → MCC 0.879

Area × per-class — additive or redundant?

Redundant. Combining both does not improve over PCT_fpr02 alone. Internal AUC rises with the filter but τ readjusts and compensates: the final result is invariant. Conclusion: ship per-class calibration as the only post-model intervention.

PCT_fpr02 + area1200 → MCC 0.913 (unchanged)

5 · Scope of this validation

What this experiment measures is DIRD+'s autonomous mode — the model operating alone against APTOS ground truth, unassisted. It's the most demanding scenario for the product, and the numbers hold up: Sens 97.8% / Spec 93.1% / MCC 0.91 at ~9 FPS on CPU, with τ stable under 5-fold CV.

What this experiment does not cover:

  • Human-in-the-loop mode: the product's primary use case. Its validation is a separate clinical study — technologists labeling with and without AI, decisions with and without the algorithm, verifier as adjudicator. Another story, planned, not part of this technical validation.
  • Single cohort: APTOS 2019 (India, heterogeneous cameras). External cross-validation on Messidor-2 / EyePACS is the next step to confirm population τ values.
  • Clinical granularity: the operational output is binary "needs attention yes/no". ICDR 0-4 staging is a distinct task, owned by the specialist.
  • Image-level metrics: APTOS provides no lesion-level annotations, so validation is reported as image-level binary classification — consistent with the use case.

In short: this validation covers the floor of the product (model alone). The ceiling — performance in real HITL flow — is measured in a dedicated clinical study.
