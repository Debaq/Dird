# Privacy Policy

**DIRD+ — Diabetic Imaging Retinopathy Detector**
Last updated: 2026-05

This document describes how DIRD+ handles patient data, what it stores, and the principles that guide its design. DIRD+ is open source software (AGPL-3.0); this policy applies to the official distribution. Forks may modify behavior — verify before deploying.

## 1. Scope

DIRD+ is a desktop and web application for AI-assisted diabetic retinopathy screening. It runs entirely on the device of the healthcare professional who operates it. **The DIRD+ project does not operate any server, cloud service, or backend that receives data from the application.** This privacy policy describes what the application does locally; the legal responsibility for patient data handling lies with the deploying institution and the operating clinician under their local jurisdiction.

## 2. Privacy principles

DIRD+ is built on the principles of **Privacy by Design** (Cavoukian, 2009):

1. **Proactive, not reactive.** Privacy controls are built into the architecture, not added as features.
2. **Privacy as the default.** No data leaves the device unless the operator explicitly exports it.
3. **Privacy embedded in design.** Local-only processing is a core architectural choice, not a configuration option.
4. **Full functionality.** Privacy does not require sacrificing usability or accuracy.
5. **End-to-end security.** Data is protected at rest with strong encryption.
6. **Visibility and transparency.** This document and the source code disclose all data handling.
7. **Respect for user privacy.** The operator and patient retain full control over the data.

## 3. What DIRD+ does NOT do

- **No telemetry.** The application does not collect usage metrics, crash reports, performance statistics, or any data about how the application is used.
- **No analytics.** No third-party analytics services (Google Analytics, Mixpanel, etc.) are embedded.
- **No cloud sync.** Patient data is never transmitted to any external server, including servers operated by the DIRD+ project.
- **No automatic updates with data exfiltration.** Application updates are user-initiated through GitHub Releases.
- **No tracking identifiers.** The application does not generate or transmit any device, user, or installation identifier.
- **No third-party data sharing.** No data is shared with any third party under any circumstances by the application.

## 4. What DIRD+ stores locally

The operating clinician may optionally enter patient information into DIRD+. **All fields are optional**; the clinician decides which fields to fill on a per-case basis (data minimization by design).

The following data, when provided, is stored locally on the device:

### 4.1 Patient data
- Patient name
- National identifier (RUT, DNI, or equivalent)
- Date of birth
- Gender
- Contact information (phone, email)
- Clinical history and prior diagnoses

### 4.2 Clinical data
- Retinal fundus images (PNG/JPEG)
- AI-generated detection results (bounding boxes, classes, confidence scores)
- Segmentations and measurements
- Clinical PDF reports generated by the application

### 4.3 Professional data
- Operating clinician identification (for traceability of who created each record)

### 4.4 Storage location
- **Structured data**: SQLite database encrypted with SQLCipher, located under the application's user data directory (`<app_data>/dird.db`).
- **Image and PDF assets**: stored as blobs within the database or as files within `.dird` export containers.
- **Exported case files** (`.dird`): generated explicitly by the operator and saved to a location of their choosing.

The location of these files depends on the operating system and the application's configuration. Patients and operators can locate, copy, or delete these files using standard file system tools.

## 5. Encryption at rest

Starting with DIRD+ v2.0:

- The local SQLite database is encrypted using **AES-256** via SQLCipher. Page-level AES-256-CBC + HMAC-SHA-512 page authentication.
- Exported `.dird` files are encrypted using **AES-256-GCM** with authenticated headers (see `docs/dird-format.md` §1.1 for the binary container layout).
- Encryption keys are derived from user-supplied passphrases using **Argon2id** with OWASP 2025 parameters (memory cost 64 MiB, time cost 3, parallelism 4; RFC 9106). The 32-byte derived key is passed directly to SQLCipher as a raw key (`PRAGMA key = x'<hex>'`), bypassing SQLCipher's internal PBKDF2 in favor of the stronger Argon2id derivation.

### 5.1 Dual-password model

DIRD+ uses two separate passwords with distinct purposes:

| Password | Purpose | Where configured |
|----------|---------|------------------|
| **Application password** | Unlocks the encrypted local SQLite database at every launch. Cannot be cached on disk. | First-launch wizard |
| **Export password** | Encrypts `.dird` archives generated for sharing, backup, or transfer. Kept in memory only during the active session. | First-launch wizard or Settings → Security |

This separation provides defense in depth: a `.dird` shared with a colleague is encrypted under a key that the colleague (and only the colleague) holds, while the local database remains protected by a different passphrase that never leaves the operator's mind.

### 5.2 Password recovery

**There is no password recovery mechanism for the encryption password.** If the operator forgets this password, the encrypted data becomes permanently unreadable. This is an intentional security trade-off (similar to Signal, Bitwarden, VeraCrypt). Operators must store the encryption password securely outside the application.

## 6. Data transmission

DIRD+ does not transmit data over the network as part of its core functionality. The only network activity initiated by the application is:

- **Manual update checks** (user-initiated): fetches the latest release information from GitHub Releases. No patient data is sent.
- **Reference model downloads** (user-initiated, from Settings → AI Models): fetches the official DIRDvN models from the public `Debaq/dird_models` repository on GitHub. No patient data is sent. Third-party models loaded via Settings → AI Models → "Add Model" are loaded entirely from the user's filesystem and never transmitted.

## 7. Patient rights and operator responsibilities

Because DIRD+ is software deployed locally, the legal responsibilities for upholding patient rights (access, rectification, deletion, portability, etc.) fall on the **deploying institution and the operating clinician**, under the applicable law in their jurisdiction.

DIRD+ provides the technical means to support these rights:

| Right | DIRD+ feature |
|-------|---------------|
| Access | View patient record within the application |
| Rectification | Edit any patient field within the application |
| Erasure | Delete patient record via the application |
| Portability | Export to `.dird` (open ZIP+JSON format) or extract raw data from SQLite |
| Restriction | Encryption password access control |

## 8. Applicable legal frameworks

DIRD+ is designed to be **compatible** with the following privacy frameworks. Compliance in any specific deployment depends on the implementing institution:

### 8.1 Chile
- **Law 19.628** on the Protection of Private Life (currently in force).
- **Law 21.719** on Personal Data Protection (enacted 2024, full enforcement December 2026), aligned with GDPR principles.

### 8.2 European Union
- **Regulation 2016/679 (GDPR)**: alignment by design through data minimization, local processing, no third-party transfers, user control over data, and encryption at rest.

### 8.3 United States
- **HIPAA Privacy Rule**: alignment by design through local-only PHI storage, no third-party disclosure, and encryption at rest (HIPAA Security Rule § 164.312(a)(2)(iv)).

### 8.4 General principles
- **OECD Guidelines on the Protection of Privacy** (1980, updated 2013).
- **ISO/IEC 27001** information security management principles.

## 9. Threat model

DIRD+ explicitly addresses the following threats:

| Threat | Mitigation |
|--------|------------|
| Network interception | All processing is local; no data transmitted. |
| Server breach | No server exists. |
| Third-party data sale | No data leaves the device. |
| Telemetry-based profiling | No telemetry. |
| Device theft (offline) | Encryption at rest (AES-256, Argon2id-derived key). |
| Unauthorized access while device is on | In-app user password required for session. |
| Forensic recovery of deleted records | Secure deletion is on the roadmap (see ROADMAP.md). |

DIRD+ does **not** address:

| Out-of-scope threat | Reason |
|---------------------|--------|
| Compromised operating system | Beyond software-layer mitigation; deploying institution's responsibility. |
| Physical access with active session | In-app session timeout planned; meanwhile, use OS screen lock. |
| Side-channel attacks on the device | Beyond typical clinical software threat model. |
| Insider misuse by authorized operator | Audit logging is on the roadmap; institutional policy is the primary control. |

## 10. Children and vulnerable populations

DIRD+ may be used by clinicians on pediatric or otherwise vulnerable patients. The same privacy protections apply. The deploying institution is responsible for obtaining informed consent (or assent, as applicable) under their local regulation.

## 11. Disclaimer

DIRD+ is **research and development software**. It is not a certified medical device. It is intended as a screening aid for qualified healthcare professionals. Clinical decisions must be made by the qualified professional, who bears the responsibility for patient care.
